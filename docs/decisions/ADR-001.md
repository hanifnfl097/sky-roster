# Architecture Decision Record (ADR)

**ADR-001:** Adopt Clean Architecture with Hexagonal (Ports & Adapters) Pattern for SkyRoster Backend
**Status:** Proposed
**Date:** 2026-03-09
**Author:** @backend-dev (Aviation Software Architect)

---

## 1. Context

SkyRoster is an enterprise-grade aviation crew management system that must:
- Enforce complex, regulation-heavy business rules (CASR 121, ICAO FTL, crew complement, document expiry).
- Support 3 different client platforms (Web Admin, Mobile App, potentially future integrations).
- Provide strong audit trails and transactional consistency.
- Be testable in isolation (business rules must be verifiable without infrastructure dependencies).
- Be maintainable by a team over multiple years as aviation regulations evolve.

The business logic is the most critical and complex part of the system. It must be isolated from frameworks, databases, and external services so that regulatory changes can be implemented and tested independently.

## 2. Decision Drivers

- **Regulatory Complexity**: FTL rules, document expiry, and crew complement logic must be expressed as pure domain logic, not tangled with HTTP controllers or database queries.
- **Testability**: Business rules must be unit-testable without spinning up Spring Boot, PostgreSQL, or any infrastructure.
- **Framework Independence**: The core domain must not depend on Spring, Hibernate, or any specific framework — only on Java and domain abstractions.
- **Long-term Maintainability**: Aviation regulations change. The architecture must allow modifying business rules without touching infrastructure code.
- **Team Scalability**: Multiple developers must be able to work on different modules (Crew, FTL, Rostering) without merge conflicts.

## 3. Considered Options

### Option A: Layered Architecture (Traditional MVC)
- **Pros**: Simple, widely understood, fast to scaffold.
- **Cons**: Business logic leaks into Controllers and Services. JPA entities become the domain model (anemic entities). Hard to test FTL rules without database. Tight coupling between layers.

### Option B: Clean Architecture (Hexagonal / Ports & Adapters)
- **Pros**: Domain layer is completely isolated (zero framework dependencies). Ports define contracts, adapters implement them. Business rules are pure Java, fully unit-testable. Framework changes (e.g., switching from JPA to jOOQ) don't touch domain code.
- **Cons**: More boilerplate (port interfaces, adapter classes, DTO mapping). Steeper learning curve for junior developers. Over-engineering risk for simple CRUD modules.

### Option C: Domain-Driven Design (DDD) with CQRS + Event Sourcing
- **Pros**: Best for complex domains with event-driven workflows. Full audit trail via event log.
- **Cons**: Very high complexity. Event sourcing adds significant infrastructure overhead. Over-engineering for this project's scale. Steep learning curve.

## 4. Decision

> We chose **Option B: Clean Architecture (Hexagonal / Ports & Adapters)** because the aviation domain has regulation-heavy business rules that MUST be isolated, testable, and framework-independent. The additional boilerplate is a worthwhile trade-off for long-term maintainability and regulatory compliance confidence.

We will apply DDD concepts (Aggregates, Value Objects, Domain Services) within the Clean Architecture layers where appropriate, but without the full CQRS/Event Sourcing complexity.

### Key Rules
1. **Domain layer has ZERO imports from `org.springframework.*` or `javax.persistence.*`**.
2. **JPA Entities are NOT domain models**. They live in `adapter.outbound.persistence.entity` and are mapped to/from domain models.
3. **All business rules** (FTL, crew complement, document expiry) live in `domain.service`.
4. **Controllers** never contain business logic — they only delegate to use cases.
5. **Dependency direction**: Adapters → Application → Domain. Never the reverse.

## 5. Consequences

### Positive
- FTL rule engine can be fully unit-tested with plain JUnit (no Spring context).
- Aviation regulation changes are isolated to `domain.service` — zero risk to infrastructure code.
- Multiple adapters can be added (e.g., GraphQL, gRPC) without touching domain.
- Strong separation of concerns enables parallel development by team members.

### Negative
- More files and boilerplate compared to traditional MVC (~30% more classes).
- DTO ↔ Domain ↔ JPA Entity mapping adds code volume (mitigated by MapStruct).
- Junior developers may need onboarding to understand the architecture.

### Risks & Mitigations
| Risk | Mitigation Strategy |
| :--- | :--- |
| Over-engineering simple CRUD modules (Fleet, Airports) | Allow "lean adapters" — skip use case interface for trivial CRUD, but still maintain package structure |
| Mapping fatigue (3 layers of models) | Use MapStruct for automated mapping; maintain clear naming conventions |
| Team unfamiliarity with hexagonal architecture | Provide architecture documentation, code examples, and pair programming sessions |
